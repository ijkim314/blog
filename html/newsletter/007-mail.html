GitHub Actions로 React 프로젝트 자동 배포하기

개발 결과물을 배포하고, 접속 URL을 공유하는 방법에 대해 얘기하려고 합니다.
vite 환경 기준 개인 프로젝트에서는 Netlify를 사용했지만, 이번에는 GitHub Actions와 GitHub Pages를 연결해 보기로 했습니다. 코드를 GitHub에 올리면 빌드와 배포가 자동으로 실행되므로 더 빠르고 일관되게 결과물을 공유할 수 있습니다.

설정 과정은 다음과 같습니다.

  1. 패키지 매니저와 의존성 설치 방식 통일
    저장소의 잠금 파일에 맞는 명령어를 사용합니다. package-lock.json이 있다면 npm i, yarn.lock이 있다면 yarn i을 사용합니다.
  2. GitHub Pages 배포 방식 선택
    저장소의 Settings → Pages로 이동한 뒤, Build and deployment의 Source를 GitHub Actions로 선택합니다.
  3. GitHub Actions 워크플로 작성
    .github/workflows/deploy.yml 파일을 만들고 체크아웃, Node.js 설정, 의존성 설치, 빌드 순서로 작업을 구성합니다. 빌드가 끝나면 dist 폴더를 Pages 아티팩트로 업로드한 뒤 배포합니다.
    이 부분은 AI 이용하면 쉽게 작성이 가능합니다.
  4. 빌드 명령어 설정
    package.json의 scripts에 "build": "tsc -b && vite build --mode production"을 추가합니다. 워크플로에서는 선택한 패키지 매니저에 맞춰 npm run build 또는 yarn build를 실행합니다.
  5. 배포 경로 지정
    vite.config.ts에서 base: "/clims-fe/"로 설정해 GitHub Pages의 저장소 경로와 맞춥니다. 이 값은 https://사용자명.github.io/clims-fe/처럼 하위 경로에 배포할 때 필요합니다. 사용자 사이트나 사용자 지정 도메인에 배포한다면 기본값인 /를 사용합니다.
  6. 배포 결과 확인
    GitHub의 Actions 탭에서 워크플로가 성공했는지 확인한 뒤, Pages 화면에 표시된 URL로 접속해 최종 결과를 점검합니다.
배포 전에 꼭 확인하세요
GitHub Actions는 기본적으로 GitHub Pages에 파일을 배포할 권한이 없습니다. 따라서 워크플로 파일에 pages: write와 id-token: write를 설정해야 합니다.
pages: write는 빌드 결과물을 GitHub Pages에 올릴 수 있는 권한이고, id-token: write는 GitHub가 허용된 워크플로의 배포 요청인지 확인할 때 사용하는 권한입니다. 이 설정이 빠지면 빌드는 성공해도 배포 단계에서 실패할 수 있습니다.

이 권한은 프로젝트 루트의 .github/workflows/deploy.yml 파일에서 jobs와 같은 최상위 위치에 permissions 항목으로 추가합니다. 저장소의 코드를 가져오는 데 필요한 contents: read도 함께 설정합니다.
permissions:
  contents: read
  pages: write
  id-token: write
빌드와 배포를 서로 다른 작업(job)으로 나눴다면 배포 작업에 needs: build도 추가합니다. 이는 “빌드가 성공한 뒤에 배포를 시작하라”는 의미입니다. 빌드와 배포를 하나의 작업에서 순서대로 실행한다면 이 설정은 필요하지 않습니다. needs: build는 아래처럼 deploy 작업 안에서 runs-on과 같은 들여쓰기 위치에 작성합니다.
jobs:
  build:
    # 빌드 단계

  deploy:
    needs: build
    runs-on: ubuntu-latest
    # 배포 단계

한 번 설정해 두면 이후에는 코드를 업데이트할 때마다 같은 과정으로 자동 배포할 수 있습니다. 반복 작업은 줄고, 팀원에게 최신 결과를 공유하는 과정도 훨씬 간단해집니다.
